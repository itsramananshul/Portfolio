01
ARIA — Adaptive AI System ↗

aria.anshulraman.com · Live in production · Built entirely solo

Most AI systems answer questions. ARIA decides how to engage with you — reading emotional state, detecting behavioral patterns, and adapting in real time. Not a wrapper. A system.

Node.jsPythonTypeScript VercelSupabaseGemini 2.5 Groq WhisperTelegram BotPWA

By the numbers

12 serverless functions — at the hard ceiling of the platform, every slot used
5 long-running edge functions for tasks that exceed serverless timeouts
6 scheduled autonomous jobs, running as frequently as every 15 minutes
4 intelligence phases, each feature-flagged and shadow-validated before activation
16/16 automated regression tests passing against the live production endpoint
0 times you have to open an app for it to be working

What it does differently

  • Four-phase behavioral intelligence pipeline. Signal observation → thinking layer → pattern memory → micro-coaching. Each phase ships dormant, runs in shadow against real traffic, and activates only when accuracy is confirmed. Circuit breaker auto-disables any phase on failure.
  • Per-turn emotional signal detection. Stress, avoidance, decision paralysis, overload, uncertainty — each confidence-scored. Behavioral directives shape how ARIA engages, not what it says. Per-mode cooldowns, global throttle, self-harm pre-exit, and silent fallback on any error.
  • Six-layer memory architecture. Permanent profile (distilled every 72h), living memory (appended every conversation), active document context (confidence decay/boost/prune per turn), cold memory with recency weighting, per-person people graph with sensitive-fact gates, and full-text session log search.
  • A system that knows when to stop. Topic sovereignty layer detects dismissals in any natural phrasing, stores scoped suppressions across all output surfaces — briefings, alerts, follow-ups — without deleting the underlying memory.
  • Hybrid knowledge retrieval over a full personal corpus. Semantic and keyword search run in parallel, merged, deduplicated, reranked. 15-minute re-indexing with SHA-based change detection. Source routing that learns preferred retrieval paths from outcome data. Hourly semantic clustering across the entire vault.
  • Autonomous execution engine. Up to 10 steps, dispatched via edge function to bypass serverless limits. Per-step state persists to the database — mid-run kills are survivable. Output sanitization rejects scaffolding tokens before the user sees the response.
  • Non-AI final gate for irreversible actions. The last check before any email sends is hardcoded deterministic logic — not a language model. Defense in depth only works when both layers can fail independently.
  • Always-on desktop voice agent. Wake word, local STT, neural TTS. 1.2s post-speech delay to prevent self-triggering. Automatic response compression for voice: 1–3 sentences, no markdown.
  • Production engineering. 16-test regression suite + multi-scenario simulation suite. Unit tests for every module. SQL migrations. Point-in-time snapshots with restore scripts. Feature flags on every behavioral change.

02
Auriva AI — Multi-Faith Spiritual Guide ↗

auriva.anshulraman.com

Most spiritual apps pick one religion. Auriva chose all of them — 16 sacred traditions, each with a dedicated live scripture API, an individually authored Gemini system prompt, and a voice conversation agent engineered differently for iOS, Android, and desktop. The differentiator is not breadth; it’s that every tradition is genuinely first-class: Arabic Quranic text from a live endpoint, Gurmukhi from a Sikh gurbani service, and a two-model hybrid for Bible where Gemini picks the verse reference and a scripture API fetches the text.

JavaScriptVercel EdgeSupabase Gemini 2.5Web Speech APIVoice AI API.BibleMulti-Faith AI
  • Verse retrieval runs a 3-tier cascade: 4 live scripture APIs at tier one (GurbaniNow, alQuran.cloud, VedicScriptures, API.Bible), Gemini generation with per-source prompts at tier two, and hardcoded verse pools at tier three — zero UI errors regardless of network state.
  • Two-phase VAD runs 2 calibration passes before each desktop voice session — measures ambient silence before TTS fires, then re-measures speaker bleed while TTS plays, deriving a threshold that blocks self-triggering without cutting off a real user speaking over the AI.
  • Android TTS sentence-chunking splits every response into individual sentences chained as separate speech objects via callbacks, working around the OS hard utterance cutoff; each sentence carries its own safety timeout.
  • iOS audio unlock fires a near-silent utterance inside the button-click gesture handler before voice mode activates, bypassing the browser’s restriction on speechSynthesis in async contexts.
  • Bible verse pipeline is two-stage: Gemini runs a high-temperature reference-selection pass across all 66 books, then API.Bible retrieves the actual text with a three-translation waterfall fallback (NIV → KJV → AMP).
  • Gemini SSE edge proxy on Vercel Edge Runtime parses the stream and forwards raw text tokens — first words reach the client before the full response is computed.
  • Message editing truncates in-memory history at the selected index, wipes and atomically re-inserts the Supabase messages table, and restores the original text into the input for re-submission.
  • Dual-path auth boot runs getSession and onAuthStateChange in parallel with a readiness guard specifically for Safari’s post-redirect timing issues on older iOS.
  • No framework — auth screen, onboarding flow, and full chat interface hand-built in vanilla ES modules.

Stack

Gemini 2.5 Flash · Vercel Edge Functions · Supabase (Auth + Postgres + RLS) · Vanilla ES Modules · Web Speech API · API.Bible · alQuran.cloud · GurbaniNow · VedicScriptures
