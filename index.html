01
ARIA — Personal AI Operating System ↗

aria.anshulraman.com · Live in production · Built entirely solo

Most AI projects are demos. ARIA is not. ARIA is a fully autonomous personal AI system I built from scratch. It runs continuously in the background — monitoring, reasoning, writing, and acting — 24/7, without being asked. It knows my grades, my deadlines, my conversations, my habits, and my goals. It manages them automatically. Not a chatbot. Not a wrapper. A system.

Node.jsPythonTypeScript VercelSupabaseGemini 2.5 Groq WhisperTelegram BotPWA Three.js

By the numbers

12 serverless functions in production — at the hard ceiling of the hosting plan, every slot used
6 scheduled autonomous jobs, running as frequently as every 15 minutes
5 long-running edge functions handling tasks too complex for serverless timeouts
16/16 automated regression tests passing against the live production endpoint
3 interfaces — Telegram, a PWA, and an always-on voice agent on desktop
0 times you have to open an app for it to be working

What makes it technically interesting

  • The hardest part of building an AI system that operates continuously is not the AI — it’s the judgment layer. ARIA has to decide, on every single interaction, what kind of question it’s dealing with, what data source is actually relevant, and what level of action is safe to take. Getting that wrong has real consequences: a misclassified question sends the user into a multi-step autonomous research loop instead of giving an immediate answer. I built, tested, and iterated on that judgment layer extensively — validating every change against real production traffic before it touched live behavior.
  • The memory system required a different kind of thinking. There are things ARIA should remember forever, things it should remember temporarily, and things it should stop surfacing once you’ve told it they’re resolved — all without deleting anything. Designing those three layers to work together without conflicting took more architectural iteration than most of the features people would notice.
  • Destructive actions — sending an email, modifying a calendar event — have a dedicated safety architecture that is explicitly not AI-based. The last gate before any irreversible action executes is deterministic, hardcoded logic. I made that decision deliberately: defense in depth only works if the two layers can fail independently of each other.
  • The behavioral intelligence layer — the part that detects when I’m overwhelmed, stuck in a loop, or avoiding something, and adjusts how ARIA engages — was built in phases, shipped dormant, validated in shadow against real traffic, and only activated once the data confirmed it was behaving correctly. That pattern is how I’d handle any high-risk change in a production system.

EV3 Autonomous Robot — UC Engineering Design Project ↗

Fully autonomous robot that navigates a line track, detects and grabs a bin by weight classification, and delivers it to the correct drop zone. Built from scratch with zero prior hardware experience.

LabVIEWLEGO Mindstorms EV3Sensors State MachineENED1002C
  • Single LabVIEW While Loop with a GRabbed boolean state machine controlling every behavior in real time — line following, bin detection, claw actuation, carry, and drop-off — no sequential structures, no human input.

  • 45 documented test runs across 3 experimental factors — surface color, lighting conditions, and tape contrast — with full descriptive statistics to validate sensor reliability before demo day.
  • Applied the full engineering design process: stakeholder analysis, morphological charts across 6 design functions, weighted decision matrices, and descriptive statistics — not just built it, actually engineered it.
